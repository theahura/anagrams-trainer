# Noridoc: games/speedrun/src

Path: @/games/speedrun/src

### Overview

- Runtime source modules for the Speedrun game, organized as pure-logic ES modules plus two DOM-touching modules (`main.js` for game loop/lifecycle, `renderer.js` for Canvas 2D drawing)
- All game state is plain mutable objects (player, timer, level, inputState) -- no reactive framework, no classes

### How it fits into the larger codebase

- `@/speedrun/index.html` loads `main.js` as the entry point via `<script type="module">`
- `@/tests/speedrun/` imports directly from these modules for unit and integration testing
- `@/games/speedrun/style.css` (sibling directory) styles the HTML shell and results overlay that `main.js` manipulates via DOM
- The PRNG in `prng.js` is a standalone copy of the cyrb128+sfc32 pattern also used in `@/games/reword/src/prng.js`
- `firebase.js` and `leaderboard.js` depend on the `firebase` npm package (client SDK, modular imports). `nameFilter.js` depends on `leo-profanity`. Both are declared in `@/package.json`

### Core Implementation

- **`game.js`** -- Game lifecycle module holding `startRun(timer)` and `restartRun(player, level, timer)`. `startRun` sets `timer.running = true` without touching input state -- this is deliberate so the keypress that triggers game start also executes as a movement action (jump/dash) on the first frame. `restartRun` resets player (via `resetPlayer`), clears all coin `collected` flags, and resets the timer to zero with `running = false`. Exists as a separate module from `main.js` so game lifecycle logic is testable without DOM side effects
- **`main.js`** -- Entry point. Creates all game objects on DOMContentLoaded, runs `requestAnimationFrame` loop, manages three-state lifecycle (READY/PLAYING/COMPLETE), builds the results overlay DOM on completion. All reset paths (mid-run R key, post-completion R/Enter/button) transition to READY state and call `updateSettingsButtonState()` to re-enable the settings gear. `startGame()` is the single function that transitions READY to PLAYING, delegating to `startRun(timer)` from `game.js`. It does not clear frame input -- this is intentional so the edge-triggered input (`jumpPressed`/`dashPressed`) that caused the game start is preserved and consumed by the first frame's physics update. A persistent `handleKeyDown()` listener enables R key restart (mid-run or from results) and Enter key restart (from results only). Mid-run R restart increments the attempt counter and saves stats before resetting. Keyboard restart handling is deliberately separate from the movement input system in `input.js`. Also wires up path recording (record each frame, reset on restart), ghost interpolation (reads PB path for selected category from stats), the settings UI (gear button opens modal, radio buttons for ghost category, modal blocked during PLAYING), the how-to-play modal (via `setupHowToPlayModal()` from `howToPlay.js`, stored in `howToPlay` variable for keyboard guard access), and the leaderboard UI: a submit-to-leaderboard flow on the results screen (name input form -> per-category Firestore writes), the leaderboard modal (tabbed view with rank display), and keyboard guards that suppress game input when the name input is focused, settings are open, the how-to-play modal is open, or the leaderboard modal is open. The keyboard guard priority chain in `handleKeyDown` is: input focus > settings open > how-to-play open (Escape closes) > leaderboard open (Escape closes) > game state handling
- **`physics.js`** -- Exports `createPhysicsConfig()` (derived constants) and `updatePlayer()` (the per-frame simulation). Handles horizontal acceleration/deceleration, jumping (with coyote time + input buffering), variable jump height, wall sliding, wall jumping with control delay, corner vaulting, dash, sprint, gravity, AABB collision resolution (X then Y), coin collection, goal detection, and off-bottom respawn. Wall jump uses two key parameters: `wallJumpHorizontalSpeed` (100px/s) controls how far the player is pushed away from the wall, and `wallJumpControlDelay` (0.04s) controls how long player input is overridden with the away-from-wall direction. These values are tuned for Hollow Knight-style repeated same-wall jumping -- the brief displacement (~7px) lets players immediately hold back toward the wall to re-cling and climb vertically. The timer is ticked each frame, cleared on grounding, and set alongside `wallJumpForceDir` when a wall jump fires. Corner vault has two triggers: (1) wall-slide vault via `isWallVaultEligible()` -- when the player's top half clears a corner while bottom half is still on the wall, jumping launches forward over the platform with `vaultMomentumBoost` (150px/s); (2) edge vault via `detectEdgeVault()` -- when the player just landed on a platform edge (groundedTime < jumpBufferTime) with their hitbox overhanging the air and pressing into the platform. The jump priority chain is: edge vault (just landed + at edge) > regular jump (grounded/coyote) > wall vault (wallDir + top clear) > wall jump > nothing. Dash initiates when `dashPressed` fires and `dashCooldownTimer` is expired: it saves current velocity into `dashSavedVx`/`dashSavedVy`, then overrides velocity to produce a pure cardinal-direction movement. If `input.down` is held without left/right, it performs a down-dash (vx=0, vy=dashSpeed). Otherwise it performs a horizontal dash (vx=dir*dashSpeed, vy=0), where dir comes from left/right input or falls back to current vx sign. Horizontal input takes priority over down when both are held. There is no up-dash. While `dashTimer > 0`, both the horizontal movement block and the vertical physics block (gravity, wall sliding, fall clamping) are skipped, freezing the player on the dash trajectory. When `dashTimer` expires, `dashSavedVx`/`dashSavedVy` are restored -- the player resumes with the velocity they had before the dash. This makes the dash a discrete displacement that does not carry momentum from other axes. Sprint is simpler: when `input.dash` (E held), `effectiveMaxSpeed` uses `sprintMaxSpeed` (375px/s) instead of `maxRunSpeed` (250px/s); the player still accelerates normally to the higher cap. Sprint does not affect vertical physics. Both dash timers and saved velocity fields are reset on off-bottom respawn alongside the other player state resets
- **`level.js`** -- Exports `generateLevel(seed)` and `getDailySeed(date)`. Builds a 25x19 tile grid using a lane-based system designed to create distinct high and low routes through the level:
  - Three lane bands divide the grid vertically: HIGH_LANE (rows 3-7), MID_LANE (rows 8-10), LOW_LANE (rows 11-15). Ground occupies the bottom two rows with random pits
  - `generateLaneChain()` places platform chains left-to-right within a lane, constrained by jump height/distance constants (JUMP_H=4, JUMP_D=5) so consecutive platforms are always reachable
  - `placeConnectors()` adds mid-lane platforms at columns between overlapping high/low platforms, allowing players to transition between routes
  - Coin placement is route-biased: red coins are placed on high-lane platform surfaces (`collectSurfacePositions()`), blue coins on low-lane platforms plus ground-level positions. This means 100% red and 100% blue completion categories require different paths. Both position pools are filtered by `isOutsideGoal()` before shuffling, which excludes any position within one TILE_SIZE of the goal center in each axis -- this prevents coins from spawning inside the goal hitbox where they would be uncollectable (the goal triggers run completion before the player can pick them up)
  - Goal is placed on the highest right-half high-route platform; start is on the ground at the left. Goal placement must happen before coin placement because the coin filter depends on the goal's coordinates
  - BFS reachability check + `addBridgePlatforms()` fallback is retained as a safety net. The level object shape is unchanged from previous versions
- **`player.js`** -- Exports `createPlayer(x, y)`, `resetPlayer(player, x, y)`, and dimension constants (`PLAYER_WIDTH=20`, `PLAYER_HEIGHT=28`). Player state is a plain object with position, velocity, grounded/wall flags, jump timers, wall jump control state (`wallJumpControlTimer`, `wallJumpForceDir`), dash timers (`dashTimer`, `dashCooldownTimer`), dash velocity save slots (`dashSavedVx`, `dashSavedVy`), coin counts, and goal flag. Both `createPlayer` and `resetPlayer` initialize the wall jump control and dash fields to zero
- **`input.js`** -- Exports `createInputState()`, `setupInputListeners(inputState)`, `clearFrameInput(inputState)`. Maps arrow keys, WASD, and spacebar to `left`/`right`/`down`/`jump`/`jumpPressed` state, and E key to `dash`/`dashPressed`. The `down` input (ArrowDown/s/S) currently only affects dash direction (selecting down-dash when held without left/right). Both `jumpPressed` and `dashPressed` use the same edge-detection pattern: set on keydown only if the held state is not already true, cleared each frame by `clearFrameInput()`
- **`renderer.js`** -- Exports `createRenderer(canvas)` which returns `{ render(level, player, timer, state, stats, daySeed, ghostPos, currentPath) }`. Draws tiles, goal with glow, coins as circles, player as rectangle, and HUD. During PLAYING, also draws a ghost player (semi-transparent blue rectangle) at the interpolated PB position if `ghostPos` is provided. During COMPLETE, draws a path line overlay from the `currentPath` array. When `player.dashTimer > 0`, the player is rendered white (`#ffffff`) with a semi-transparent afterimage trail: for horizontal dashes the trail is offset 12px behind (opposite to `vx` direction), for down-dashes (vy > 0) the trail is offset 12px above the player. The HUD shows: day seed in top-right (on both READY and PLAYING screens), "press any key" prompt on READY, and during PLAYING: timer, coin counts, PB any% time, and attempt count
- **`timing.js`** -- Exports timer creation/update, `formatTime(seconds)` (M:SS.mmm format), and `createCompletionRecord()` which produces `{ anyPercent, hundredRed, hundredBlue, hundredPercent }` where category times are null if the required coins were not all collected. `hundredPercent` requires ALL red AND ALL blue coins -- a single run collecting every coin qualifies for all four categories simultaneously
- **`stats.js`** -- Exports `loadStats(seed)`, `saveStats(seed, stats)`, `updatePersonalBest(stats, record, paths)`. localStorage key pattern: `speedrun-stats-{seed}` (e.g. `speedrun-stats-2026-04-08`). Tracks four category PBs: `bestAnyPercent`, `bestHundredRed`, `bestHundredBlue`, `bestHundredPercent`. Personal bests update only if the new time is strictly faster. The optional `paths` parameter stores path data alongside PB times -- each category's path is only updated when that category's time improves. `updatePersonalBest` handles old stats objects that lack `bestPaths` or `bestHundredPercent` by falling back to null-initialized defaults, making the new category backward compatible with existing localStorage data- **`path.js`** -- Exports `createPathRecorder`, `recordFrame`, `resetRecorder`, `getPath`, `isPathComplete`, `interpolatePosition`. The recorder samples player position as `[x, y, t]` tuples at a configurable interval (default 0.05s / ~20Hz). `isPathComplete` returns false if the path is empty or the final sample exceeds `maxDuration` (default 30s). `interpolatePosition` performs linear interpolation between adjacent samples for a given elapsed time, returning null if elapsed exceeds the path's time range
- **`settings.js`** -- Exports `loadSettings()` and `saveSettings(settings)`. Persists to localStorage under `speedrun-settings`. Currently stores `ghostCategory` (one of `off`, `anyPercent`, `hundredRed`, `hundredBlue`, `hundredPercent`; defaults to `anyPercent`)
- **`firebase.js`** -- Initializes the Firebase app with a static config and exports the Firestore `db` instance. Uses modular imports (`firebase/app`, `firebase/firestore`) for tree-shaking
- **`leaderboard.js`** -- Firestore CRUD for the global leaderboard. `submitScore` validates category (must be one of `anyPercent`/`hundredRed`/`hundredBlue`/`hundredPercent`), time (finite number, 0 < t <= 3600), and name (via `validateName`) before writing to `leaderboards/{daySeed}/scores`. `fetchLeaderboard` queries by category ordered by time ascending with a limit (default 50). `fetchPlayerRank` uses `getCountFromServer` to count documents with a faster time, returning rank as count + 1
- **`howToPlay.js`** -- How-to-play modal module. Exports `hasSeenHowToPlay()`, `markHowToPlaySeen()`, and `setupHowToPlayModal()`. Uses localStorage key `speedrun-seen-how-to-play` (value `'1'`) for first-visit detection -- same pattern as Reword's how-to-play. `setupHowToPlayModal()` queries DOM elements (`#how-to-play-modal`, `#how-to-play-btn`, `.htp-close`, `.htp-controls`), populates control descriptions via innerHTML, wires click handlers (open button, close button, backdrop click), auto-opens on first visit, and returns `{ isOpen: () => boolean, close: () => void }` for external keyboard guarding. The `?` button is not gated to any game state -- players can check controls at any time
- **`nameFilter.js`** -- Player name validation: 3-12 characters, alphanumeric plus underscores only, profanity-filtered via `leo-profanity`. Also manages saved player name in localStorage under `speedrun-player-name` so returning players don't re-enter their name
- **`prng.js`** -- Exports `getSeededRng(seed)`. Uses cyrb128 to hash a string seed into four 32-bit values, then sfc32 as the generator returning floats in [0, 1)

### Things to Know

- State ownership: `main.js` owns all state objects and passes them into pure functions. No module maintains its own mutable state (except input listeners mutating the shared `inputState` object). The path recorder and settings objects follow the same pattern -- owned by `main.js`, passed to pure functions in `path.js` and `settings.js`
- The `updatePlayer()` function in `physics.js` is the largest function -- it handles the entire simulation step including movement, collision, coin pickup, goal check, and respawn. This is intentional: all physics side effects happen in one place per frame
- `createCompletionRecord()` uses the same elapsed time for all categories -- there is no split timing. A category time is recorded only if the required coins were collected before reaching the goal. The `hundredPercent` category requires both `allRed && allBlue`, so it is the conjunction of `hundredRed` and `hundredBlue`
- `updatePersonalBest()` returns a new stats object (spread copy) rather than mutating in place
